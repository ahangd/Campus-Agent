# 新手教程（从 0 到 1 看懂本项目）

本教程面向第一次接触 Next.js + Dify 的同学，目标是让你能：
- 跑起来项目
- 看懂“表单 -> 对话”的完整流程
- 知道常见报错怎么排查
- 知道后续该从哪里改功能

---

## 1. 这个项目在做什么

这是一个“校园智能咨询助手”前端，核心流程是：

1) 用户先填写调研表单（学号、姓名等）  
2) 点击开始对话进入聊天页  
3) 前端调用你自己的 API 路由（不是直接打 Dify）  
4) 你的 API 路由再代理转发到 Dify  
5) 把 Dify 的流式回答实时显示在聊天窗口

---

## 2. 先跑起来

### 2.1 安装依赖

```bash
pnpm install
```

### 2.2 配置环境变量

在项目根目录创建/修改 `.env.local`：

```env
DIFY_API_KEY=你的Dify应用密钥
DIFY_API_BASE_URL=https://api.dify.ai/v1
```

说明：
- 代码里会优先读取 `DIFY_EV_AGENT`，没有就读 `DIFY_API_KEY`。
- 常见错误 `请求失败 (500)` 多数是 key 不对或未配置。

### 2.3 启动项目

```bash
pnpm dev
```

打开 `http://localhost:3000`

## 4. 核心流程讲解

## 4.1 首页与快捷提问（`hero-section.tsx`）

- 左侧“快捷提问”按钮点击后会先检查本地缓存里是否有学号和姓名。
- 有：把问题存到 `dify-ev-pending-query`，跳转 `/chat`。
- 没有：提示“请先填写学号和姓名”，并滚动到表单区域。

你会看到这几个本地键：
- `dify-ev-user-input-form`：表单草稿（实时保存）
- `dify-ev-chat-session`：提交后的表单快照
- `dify-ev-pending-query`：待自动发送的快捷问题

## 4.2 表单页（`components/dify-entry-modal.tsx`）

页面加载后会请求 `/api/dify/parameters`，拿到 Dify 的 `user_input_form` 动态渲染字段。

关键逻辑：
- 输入时实时保存草稿（防刷新丢失）
- 提交时统一校验必填
- 校验通过后：
  - 保存草稿和快照
  - 跳转 `/chat`

## 4.3 聊天页（`components/dify-chat-panel.tsx`）

发送消息时会请求 `/api/dify/chat`，后端再请求 Dify `chat-messages`。

关键点：
- `conversation_id` 会自动维护，实现多轮上下文
- 使用 SSE 流式读取，边到边显示
- `<think>...</think>` 会被清洗，不直接展示内部推理
- 思考过程单独折叠展示（在结果上方）

## 4.4 后端代理（`app/api/dify/chat/route.ts`）

为什么要代理？
- API Key 不能放在前端
- 统一做参数校验、错误处理、日志观察

当前路由做了：
- 读取并校验请求体
- 转发到 Dify
- 透传上游错误内容
- 以流式响应返回给前端

## 5. 历史记录怎么工作的

当前实现是“前端本地历史”：
- 存储键：`dify-ev-chat-history`
- 每轮会话稳定后自动保存
- 历史弹窗可点击恢复旧会话

优点：实现快、无需新后端表  
局限：换设备/换浏览器看不到历史（因为是本地存储）

## 6. 常见报错与排查

## 6.1 `请求失败 (500)`
- 检查 `.env.local` 是否正确
- 检查 Dify Key 是否失效
- 重启 `pnpm dev` 让新环境变量生效

## 6.2 `invalid_param: student_id is required`
- 说明 Dify 应用要求 `inputs.student_id`
- 确认你先填了表单并成功提交再进聊天

## 6.3 webpack cache rename ENOENT（Windows）
- 你项目已把开发缓存切为 memory，通常可避免
- 仍异常时可删除 `.next` 后重启

## 7. 你下一步最建议做什么

按优先级建议：

1) 引入 Supabase Auth 登录  
2) 把“历史记录”从本地改成服务端持久化  
3) 给聊天接口加更完善的错误码映射  
4) 给调研表单增加字段级说明（比如学号格式）
