# 新手阅读指南

这份文档是写给第一次接触这个项目的人看的。

目标不是一次性看懂所有实现细节，而是先建立一个稳定的阅读顺序：

1. 这个项目要解决什么问题
2. 页面是怎么串起来的
3. 聊天消息是怎么发出去的
4. 会话和身份信息是怎么保存在本地的
5. 结果和证据为什么会这样显示

---

## 一句话理解项目

这是一个接入 Dify 的校园咨询前端。

用户先填写身份信息和表单，再进入聊天工作台。聊天页会：

- 调用 Dify 对话接口
- 流式接收回答
- 保存本地会话历史
- 展示证据片段
- 在右侧 `Agent Monitor` 中展示过程数据

---

如果你是新手，不要一上来就看最大的文件。

按下面顺序读，会轻松很多。

### 第 1 步：先看页面入口

先看这几个文件：

- 聊天页真正的核心组件是 `components/dify-chat-panel.tsx`

### 第 2 步：看聊天工作台主流程

然后看：

- [components/dify-chat-panel.tsx](/c:/zyh毕业设计/find-best-ev-main/components/dify-chat-panel.tsx)

这里是整个工作台的核心调度层。

建议你重点看这些部分：

1. `useState`
2. `sendQuery`
3. `openHistorySession`
4. `removeHistorySession`
5. `persistCurrentSession`
6. 返回的 JSX 结构

这里建议你再额外留意两个变量：

- `threadId`
- `localSessionId`

因为这个面板不只是管理本地状态，它还要和 URL 里的 `thread` 参数同步。
前面修过一个很典型的问题：点击“新建对话”后，本地会话虽然已经重置，但旧的 `threadId` 可能会在下一拍又同步回来，结果把新会话重新覆盖掉，看起来就像按钮没反应。

所以你读这段时，可以顺手关注：

- 主动新建会话时，本地状态怎么先切换
- 路由什么时候再切过去
- 在路由真正更新前，代码怎么避免旧 thread 把状态拉回去

你不需要一开始就看懂所有 helper。

先只回答下面几个问题：

- 聊天输入从哪里来
- 点击发送后会发生什么
- 会话历史从哪里来
- Agent Monitor 用的最新消息从哪里来

读完这一步，你应该知道：

- 这个文件主要负责“流程编排”
- 它不再负责所有细节实现

### 第 5 步：看消息解析和证据格式化

再看两个和“内容处理”有关的文件：

- [lib/dify-chat-message.ts](/c:/zyh毕业设计/find-best-ev-main/lib/dify-chat-message.ts)
- [lib/dify-observation.ts](/c:/zyh毕业设计/find-best-ev-main/lib/dify-observation.ts)

这是很多新手第一次会觉得“有点绕”的地方。

建议你分开理解：

#### `lib/dify-chat-message.ts`

负责：

- 清理回答中的 `<think>` 标签
- 从 thought log 里提取工具调用
- 计算反思评分
- 找到最后一条 assistant 消息

#### `lib/dify-observation.ts`

负责：

- 解析工具返回的结果
- 拆双层 JSON
- 把字段名转成中文
- 把数组 / 对象整理成更适合展示的文本

读完这一步，你应该知道：

- 为什么页面里看到的是“课程名称、授课教师、地点”
- 为什么不是直接把原始 JSON 扔给用户

### 第 6 步：看本地会话和身份信息

接下来读：

- [lib/context-manager.ts](/c:/zyh毕业设计/find-best-ev-main/lib/context-manager.ts)
- [lib/dify-chat-state.ts](/c:/zyh毕业设计/find-best-ev-main/lib/dify-chat-state.ts)
- [lib/dify-chat-history.ts](/c:/zyh毕业设计/find-best-ev-main/lib/dify-chat-history.ts)
- [lib/dify-chat-panel-support.ts](/c:/zyh毕业设计/find-best-ev-main/lib/dify-chat-panel-support.ts)

这几份文件负责的事情分别是：

#### `context-manager.ts`

- 作为“上下文管理器”负责提取姓名和学号
- 构造聊天会话快照并恢复本地上下文
- 对外暴露统一的身份与会话恢复入口

#### `dify-chat-state.ts`

- 浏览器本地 id
- `localStorage` 读写
- 更新最后一条消息

#### `dify-chat-history.ts`

- 历史记录标题
- 历史记录按时间分组
- 历史列表增删

#### `dify-chat-panel-support.ts`

- 聊天面板内部的一些 session flow helper
- 比如欢迎消息、重置会话、删除后怎么选下一会话

这里虽然主要是 helper，但很适合带着两个具体问题去读：

- 当前会话被删除后，应该打开哪个会话
- 点击“新建对话”后，为什么不能又被旧会话覆盖回来

前一个问题主要由删除后的 next session 选择逻辑负责，后一个问题则是在 `dify-chat-panel.tsx` 里通过一次性的 thread 同步保护完成。把这两个点连起来看，你会更容易理解“本地会话状态”和“URL 会话标识”其实是两条并行但需要协调的状态线。

读完这一步，你应该知道：

- 为什么刷新后还能恢复历史
- 为什么必须先填写身份信息再聊天

## 一个最实用的阅读方法

每读一个文件，只回答三个问题：

1. 这个文件负责什么
2. 它依赖谁
3. 谁在使用它

如果这三个问题答不上来，就先不要往下钻细节。

先建立“边界感”，再看实现。

---

## 新手最容易卡住的点

### 1. 一上来就想看完整链路

不建议。

这个项目同时涉及：

- React 状态
- Next.js 路由
- 本地存储
- SSE 流式消息
- Dify 返回结构

如果一口气全看，很容易糊掉。

### 2. 把 UI 文件和逻辑文件混着看

建议分开：

- 先看页面和组件怎么摆
- 再看 `lib/` 里的处理逻辑

### 3. 想立刻修改业务功能

更好的顺序是：

1. 先能描述数据流
2. 再定位改动点
3. 最后再动代码

## 如果你只想定位一个功能

可以按这个表查：

- 表单相关：`components/dify-entry-form.tsx`、`lib/dify-entry-form.ts`
- 上下文管理器：`lib/context-manager.ts`
- 发送聊天：`components/dify-chat-panel.tsx`、`lib/dify-chat-request.ts`
- 聊天历史：`lib/dify-chat-history.ts`
- 证据展示：`lib/dify-observation.ts`
- 监视器：`components/agent-monitor.tsx`
