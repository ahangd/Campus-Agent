# 数据库设计说明

本文档用于记录当前项目在 Supabase 托管 PostgreSQL 中使用的业务表、表职责、推荐字段，以及后续扩展建议。

## 1. 当前已有数据表

根据当前数据库截图，项目已经包含以下表：

- `users`
- `courses`
- `student_courses`
- `library_info`
- `faq`
- `regulations`
- `service_process`
- `chat_records`

这套表结构已经可以支撑当前版本的校园智能助理，不需要推翻重建，重点是继续补充字段、索引、外键关系和访问控制。

## 2. 表职责说明

### 2.1 `users`

用于保存用户身份信息，当前主要承载学生身份。

建议职责：

- 保存学生基本身份信息
- 作为课表、聊天记录、服务流程的身份关联入口

建议字段：

- `id uuid primary key`
- `student_id varchar unique not null`
- `name varchar not null`
- `role varchar`
- `college varchar`
- `major varchar`
- `grade varchar`
- `class_name varchar`
- `created_at timestamptz`
- `updated_at timestamptz`

说明：

- 当前项目中前端会把“姓名”“学号”作为进入聊天前的必要身份信息。
- `student_id` 应作为最重要的业务唯一键。

### 2.2 `courses`

用于保存课程基础信息与排课信息。

建议职责：

- 提供课程名称、教师、时间、地点等查询依据
- 为“学院公共课查询”“个人课表查询”提供基础数据

建议字段：

- `id`
- `course_code`
- `course_name`
- `teacher`
- `college`
- `major`
- `grade`
- `weekday`
- `start_time`
- `end_time`
- `location`
- `building`
- `classroom`
- `week_range`
- `semester`
- `academic_year`
- `is_public`
- `created_at`
- `updated_at`

说明：

- 若当前 `courses` 已经有部分字段，不必重复新增，只需补齐缺失项。
- `weekday + start_time + end_time + location` 是课表场景里最常用的查询组合。

### 2.3 `student_courses`

用于保存学生与课程之间的选课关系。

建议职责：

- 标识某个学生选了哪些课程
- 支撑“个人课表”与“学生课程画像”

建议字段：

- `id`
- `student_id`
- `course_id`
- `semester`
- `created_at`

建议关系：

- `student_id` 关联 `users.student_id`
- `course_id` 关联 `courses.id`

说明：

- 这是个人课表场景里的核心关系表。
- 建议对 `student_id + course_id + semester` 设置唯一约束，避免重复数据。

### 2.4 `library_info`

用于保存图书馆相关信息。

建议职责：

- 存储图书馆开放时间
- 存储特殊闭馆安排或服务说明

建议字段：

- `id`
- `library_name`
- `date`
- `open_time`
- `close_time`
- `remark`
- `updated_at`

说明：

- 如果后续要支持多个馆区或自习区域，可以继续拆成更细的数据表。

### 2.5 `faq`

用于保存常见问答。

建议职责：

- 提供可直接检索的标准问答
- 用于兜底回答和常见校园问题

建议字段：

- `id`
- `question`
- `answer`
- `category`
- `tags`
- `updated_at`

说明：

- 适合收录高频静态问题，如校园卡、选课时间、图书馆借阅规则等。

### 2.6 `regulations`

用于保存规章制度文本。

建议职责：

- 存储正式制度、通知、规范类内容
- 作为证据片段的重要来源

建议字段：

- `id`
- `title`
- `content`
- `category`
- `source_department`
- `updated_at`

说明：

- 当前项目已经有证据裁剪和中文字段映射逻辑，这张表很适合作为权威依据来源。

### 2.7 `service_process`

用于保存办事流程。

建议职责：

- 存储缓考申请、报修、办卡、开证明等流程说明
- 支撑“下一步怎么做”的产品能力

建议字段：

- `id`
- `title`
- `process_type`
- `description`
- `steps jsonb`
- `source_department`
- `updated_at`

说明：

- `steps` 可以使用 `jsonb` 保存步骤列表、入口链接、所需材料等结构化信息。

### 2.8 `chat_records`

用于保存聊天记录。

建议职责：

- 保存用户与助手的消息内容
- 支撑多轮对话存档、回看、后续分析

建议字段：

- `id`
- `thread_id`
- `student_id`
- `role`
- `content`
- `thought_log`
- `query_log`
- `tool_meta jsonb`
- `created_at`

说明：

- 当前前端聊天历史仍然主要存储在本地 `localStorage`。
- 若后续要做多端同步、后台审计、对话留存，就应把 `chat_records` 作为正式落库表使用。
- 当前版本还会将评测维度埋点写入 `tool_meta`，包括 `retrievalMs`、`firstTokenAt`、`finishedAt`、`firstTokenMs`、`responseMs`、`sourceKinds` 与 `localKnowledgeHitCount`，用于实验统计与性能分析。

建议字段：

- `id`
- `building`
- `room`
- `date`
- `weekday`
- `start_time`
- `end_time`
- `status`
- `source`

- `updated_at`

说明：

### 3.2 `academic_calendar`

用途：

- 支撑“校历查询”“考试周提醒”“放假安排”等

建议字段：

- `id`
- `title`
- `event_type`
- `start_date`
- `end_date`
- `description`
- `created_at`
- `updated_at`

说明：

- 适合存储开学、选课、考试周、节假日、补课安排等时间性事件。

## 4. 当前项目的最小可用表集合

如果目标是尽快支撑当前前端与 Dify 场景，建议优先保证以下表完整可用：

1. `users`
2. `courses`
3. `student_courses`
4. `library_info`
5. `faq`
6. `regulations`
7. `service_process`
8. `chat_records`
9. `classroom_availability`
10. `academic_calendar`

其中：

- `users + courses + student_courses` 支撑身份与课表
- `library_info` 支撑图书馆问答
- `faq + regulations + service_process` 支撑知识问答与办事流程
- `chat_records` 支撑对话留痕
- `classroom_availability + academic_calendar` 支撑空教室与校历

## 5. 关系设计建议

推荐关键关系如下：

- `users.student_id` 为业务主身份键
- `student_courses.student_id -> users.student_id`
- `student_courses.course_id -> courses.id`
- `chat_records.student_id -> users.student_id`

推荐索引：

- `users(student_id)`
- `courses(course_name)`
- `courses(college, semester, weekday)`
- `student_courses(student_id, semester)`
- `chat_records(thread_id, created_at)`
- `classroom_availability(date, building, room)`
- `academic_calendar(start_date)`

## 6. 安全建议

当前截图中各表均显示为 `UNRESTRICTED`，说明暂未启用严格访问限制。

后续建议至少对以下表启用 RLS：

- `users`
- `student_courses`
- `chat_records`

原因：

- `users` 包含真实身份信息
- `student_courses` 关联学生个人课表
- `chat_records` 可能包含用户问题、历史对话、证据与工具调用信息

## 7. 与当前前端实现的关系

当前前端代码里：

- 身份信息来自表单中的“姓名”和“学号”
- 聊天会话当前主要保存在本地
- 证据片段已在前端做本地裁剪与中文字段映射

因此数据库设计要优先服务以下场景：

1. 通过 `student_id` 查个人课表
2. 通过学院或课程条件查公共课安排
3. 查图书馆开放信息
4. 查常见问答、规章制度、办事流程
5. 后续把聊天记录逐步从本地迁移到数据库

## 10. 表结构 SQL 对照

本章节提供基于当前表名的 Supabase PostgreSQL 参考 SQL，用于补字段、加索引、建关系。

说明：

- 以下 SQL 以 `public` schema 为例
- 大部分语句使用 `if not exists`
- 若你现有主键类型不是 `uuid`，执行前请先确认并调整对应字段类型

### 10.1 `users`

```sql
alter table public.users
  add column if not exists college varchar,
  add column if not exists major varchar,
  add column if not exists grade varchar,
  add column if not exists class_name varchar,
  add column if not exists created_at timestamptz default now(),
  add column if not exists updated_at timestamptz default now();

create index if not exists idx_users_student_id on public.users(student_id);
create index if not exists idx_users_role on public.users(role);
```

建议补充：

- 给 `student_id` 加唯一约束
- `student_id` 和 `name` 设为非空

### 10.2 `courses`

```sql
alter table public.courses
  add column if not exists course_code varchar,
  add column if not exists course_name varchar,
  add column if not exists teacher varchar,
  add column if not exists college varchar,
  add column if not exists major varchar,
  add column if not exists grade varchar,
  add column if not exists weekday int,
  add column if not exists start_time time,
  add column if not exists end_time time,
  add column if not exists location varchar,
  add column if not exists building varchar,
  add column if not exists classroom varchar,
  add column if not exists week_range varchar,
  add column if not exists semester varchar,
  add column if not exists academic_year varchar,
  add column if not exists is_public boolean default true,
  add column if not exists created_at timestamptz default now(),
  add column if not exists updated_at timestamptz default now();

create index if not exists idx_courses_course_name on public.courses(course_name);
create index if not exists idx_courses_college on public.courses(college);
create index if not exists idx_courses_semester on public.courses(semester);
create index if not exists idx_courses_weekday on public.courses(weekday);
```

### 10.3 `student_courses`

```sql
alter table public.student_courses
  add column if not exists id uuid default gen_random_uuid(),
  add column if not exists student_id varchar,
  add column if not exists course_id uuid,
  add column if not exists semester varchar,
  add column if not exists created_at timestamptz default now();

create index if not exists idx_student_courses_student_id on public.student_courses(student_id);
create index if not exists idx_student_courses_course_id on public.student_courses(course_id);
create index if not exists idx_student_courses_semester on public.student_courses(semester);
```

若要补外键关系，可使用：

```sql
alter table public.student_courses
  add constraint student_courses_student_id_fkey
  foreign key (student_id) references public.users(student_id)
  on delete cascade;

alter table public.student_courses
  add constraint student_courses_course_id_fkey
  foreign key (course_id) references public.courses(id)
  on delete cascade;
```

建议增加唯一约束：

```sql
alter table public.student_courses
  add constraint student_courses_unique_student_course_semester
  unique (student_id, course_id, semester);
```

### 10.4 `library_info`

```sql
alter table public.library_info
  add column if not exists library_name varchar,
  add column if not exists date date,
```

### 10.8 `chat_records`

```sql
alter table public.chat_records
  add column if not exists id uuid default gen_random_uuid(),
  add column if not exists thread_id uuid default gen_random_uuid(),
  add column if not exists student_id varchar,
  add column if not exists role varchar,
  add column if not exists content text,
  add column if not exists thought_log text,
  add column if not exists query_log text,
  add column if not exists tool_meta jsonb default '{}'::jsonb,
  add column if not exists created_at timestamptz default now();

create index if not exists idx_chat_records_thread_id on public.chat_records(thread_id);
create index if not exists idx_chat_records_student_id on public.chat_records(student_id);
create index if not exists idx_chat_records_created_at on public.chat_records(created_at desc);
```

如需对评测埋点做更高频统计，也可增加 `tool_meta` 的表达式索引：

```sql
create index if not exists idx_chat_records_tool_meta_type
  on public.chat_records ((tool_meta ->> 'type'));
```

如需补外键关系：

```sql
alter table public.chat_records
  add constraint chat_records_student_id_fkey
  foreign key (student_id) references public.users(student_id)
  on delete set null;
```

### 10.9 `classroom_availability`

推荐新增：

```sql
create table if not exists public.classroom_availability (
```

## 11. 执行顺序建议

建议在 Supabase SQL Editor 中按以下顺序执行：

1. 先补 `users`
2. 再补 `courses`
3. 再补 `student_courses`
4. 再补 `library_info / faq / regulations / service_process`
5. 再补 `chat_records`
6. 最后新增：
   - `classroom_availability`
   - `academic_calendar`

原因：

- 先补主表，再补关系表，外键更容易一次通过
- 聊天记录和扩展表放后面，回滚成本更低

## 12. 执行前检查事项

在真正执行 SQL 之前，建议先确认：

1. `courses.id` 当前是什么类型
2. `student_courses.course_id` 应该使用什么类型去关联 `courses.id`
3. `chat_records` 当前是否已有主键
4. `users.student_id` 是否已经唯一

如果现有主键不是 `uuid`，请先把对应 SQL 中的关联字段类型改成一致版本，例如 `bigint`。
